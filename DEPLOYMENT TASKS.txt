Based on the architecture of the frontend application, here is the exact list of functions and features that are currently simulated (mocked) for demonstration purposes.

1. Authentication System
Simulated Feature: User Login
How it works: The login form accepts any input (or specific demo credentials) and simulates a network request. It then generates a fake JWT token and stores it in the browser's localStorage.
Real Behavior Required: Verification against a secure user database (e.g., PostgreSQL, MongoDB) and issuance of a cryptographically signed JWT.
2. AI Chatbot (LLM Integration)
Simulated Feature: Intelligent Responses
How it works: When you send a message, the app uses a setTimeout to mimic network latency and returns a pre-written, static response string. It does not "think" or process the natural language.
Real Behavior Required: Sending the prompt to an LLM API (e.g., OpenAI GPT-4, Anthropic Claude) and streaming back the generated text.
3. Retrieval-Augmented Generation (RAG)
Simulated Feature: Source Citations
How it works: The "Sources" listed below the chat answers (e.g., "DVA Hearing Loss Guide") are hardcoded mock objects. The app is not actually searching a database or vector store to find relevant documents.
Real Behavior Required: Embedding the user query, performing a vector similarity search in a database (e.g., Pinecone, Weaviate), and retrieving actual relevant document chunks.
4. Response Curation (Feedback Loop)
Simulated Feature: Accept/Deny Buttons
How it works: Clicking "Accept" or "Deny" updates the UI state locally (e.g., changing the icon color) to show the action happened. No data is sent to a server, and the AI model is not actually retrained or adjusted.
Real Behavior Required: Sending feedback data to the backend to update the vector database metadata or trigger a fine-tuning/re-ranking process.
5. Admin Tools: Web Scraping
Simulated Feature: URL Scraper
How it works: Clicking "Scrape" in the admin panel triggers a fake progress bar or loading spinner, followed by a "Success" toast notification. It does not actually visit the URL or parse HTML.
Real Behavior Required: A background worker (e.g., using Puppeteer or BeautifulSoup) to crawl the URL, extract text, chunk it, embed it, and store it in the vector database.
6. Admin Tools: Query Management
Simulated Feature: Delete Query
How it works: Deleting a query removes it from the visible list in the React state. If you refresh the page, the query will reappear because it was not deleted from a persistent database.
Real Behavior Required: Sending a DELETE request to the API to permanently remove the record from the database.
7. Veteran Services: Questionnaire
Simulated Feature: Eligibility Calculation
How it works: Submitting the benefits questionnaire returns a hardcoded result (e.g., "30% Disability Rating") regardless of the specific answers you selected.
Real Behavior Required: A rules engine or logic service that analyzes the specific form inputs against current DVA legislation to determine actual eligibility.
8. Trust Level Scoring
Simulated Feature: Confidence Percentage
How it works: The "Trust Level" (e.g., 95%) displayed on responses is a static or randomly generated number. It does not reflect the actual semantic similarity or confidence of the retrieval system.
Real Behavior Required: Calculating a score based on the distance metric of the vector search results.
